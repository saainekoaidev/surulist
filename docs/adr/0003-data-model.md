# 0003 データモデル設計

## Status

Accepted

## Context

### 取り扱う要件

US-001〜US-006 で定義された Todo リストアプリのデータモデルを設計する必要がある。各 US が要求するデータ上の関心事を以下に整理する。

- **US-001**: 1 画面構成の Todo リスト。画面遷移なし。→ データモデルとしてはシンプルな CRUD が成立すればよい。認証画面などに起因するエンティティは不要。
- **US-002**: 「目的」を選択するリストボックスで Todo を切り替え表示。→ Todo を目的別にグルーピングする親エンティティが必要。
- **US-003**: 目的の追加・修正・削除。削除時は紐づく Todo も削除。目的ごとの Todo 件数を表示。→ 親子関係に Cascade 削除が必要。件数は集計クエリで算出可能であり、非正規化カラムは不要。
- **US-004**: Todo の新規追加。全角 40 文字程度の上限。→ テキストカラムにアプリケーションレベルのバリデーション。
- **US-005**: Todo の一覧表示 (#・進捗・Todo・登録日時・更新日時) と編集・削除。→ 連番は表示用に算出し、DB カラムにはしない。登録日時・更新日時は DB レベルで管理する。
- **US-006**: 進捗ステータスの定義。英語 (日本語) 表記。完了状態でも編集・削除可能。→ ステータスの選択肢をどこで管理するかが設計判断になる。

### 制約

- **SQLite**: ファイルベース DB。Enum 型がネイティブには存在しない。
- **Prisma 6**: ORM。SQLite との組み合わせで `enum` ディレクティブは使用不可 (Prisma の SQLite コネクタが enum を未サポート)。
- **MVP スコープ**: localhost 個人利用。マルチユーザ/認証は不要。将来の拡張可能性よりも、現時点の要件に対するシンプルさを優先する。

### 検討した代替案

#### (a) ステータスを別テーブル (StatusMaster) で管理する案

ステータスの選択肢を DB テーブルに持ち、Todo.statusId で FK 参照する方式。選択肢の追加・変更を DB 操作だけで行える利点がある。しかし、本セッションの対話で以下の理由から不採用とした。

- US-006 で定義されるステータスは 4 値 (Not Started / In Progress / Pending / Done) で固定であり、ユーザが自由に追加する要件はない。
- テーブルを増やすと JOIN が増え、CRUD の実装が複雑になる。MVP スコープに対して過剰。
- 将来ステータスを変更する必要が生じた場合は、ADR を起票して再検討すればよい。

#### (b) User エンティティを設ける案

本セッションで better-auth の導入有無を確認した結果、「認証なしで進める」と判断された。localhost 個人利用のため User テーブルは不要。

#### (c) status カラムの値形式 — snake_case vs Title Case

当初の実装で Prisma スキーマのデフォルト値を `"not_started"` (snake_case) としていたが、docs/ui.md および docs/design.md で定めた表示ラベルは `"Not Started"` (Title Case) であった。本セッションでこの不整合を検出し、「ドキュメント (ui.md / design.md) を single source of truth とする」方針に基づき Prisma スキーマ側を `"Not Started"` に修正した。DB 値と表示ラベルを一致させることで、フロントエンド側での変換処理が不要になる。

## Decision

### 採用したモデル

2 エンティティ構成: **Category** (目的) と **Todo**。

```
Category 1 ──── * Todo
```

- **Category**: US-002/003 の「目的」に対応。Todo をグルーピングする親。
- **Todo**: US-004/005/006 の Todo 項目。Category に従属し、Cascade 削除される。

### 主要な設計判断とその根拠

1. **Category → Todo の Cascade 削除を採用**
   - 根拠: US-003 で「目的を削除すると紐づく Todo も削除される」と明記されている。Prisma の `onDelete: Cascade` でこれを DB レベルで保証し、アプリケーションコードでの削除漏れリスクを排除した。

2. **ステータスは String カラム + アプリケーションバリデーションで管理**
   - 根拠: SQLite + Prisma 6 の制約で DB レベルの enum が使えない。4 値固定 (US-006) であり StatusMaster テーブルを設けるほどの可変性がない。アプリケーションコードで許容値を定数として管理し、API のバリデーションで制御する。

3. **ステータスの DB 値は Title Case (`"Not Started"` 等)**
   - 根拠: US-006 の「英語 (日本語) で出してくれると格好良い」要件に基づき、DB 値 = 表示ラベルとした。変換レイヤが不要になり実装がシンプルになる。ドキュメント (ui.md / design.md) との整合性を優先し、対話中に snake_case から修正した。

4. **Todo 件数は非正規化カラムを持たず、集計クエリ (`_count`) で算出**
   - 根拠: US-003 で目的ごとの Todo 件数表示が求められるが、Prisma の `include: { _count: { select: { todos: true } } }` で取得できる。個人利用のデータ量では集計のパフォーマンス問題は発生しない。非正規化カラムを持つと更新整合性の維持コストが増える。

5. **User エンティティは設けない**
   - 根拠: US-001 で「自分が使えればよい」、localhost 個人利用と明記されている。本セッションで認証 (better-auth) の要否を確認し、「認証なしで進める」と判断された。

6. **連番 (#) は DB カラムにせず、表示時に算出**
   - 根拠: US-005 の一覧に表示される `#` は表示上の行番号であり、データとしての意味を持たない。Todo の追加・削除で欠番が生じるため、永続化せずフロントエンドで配列インデックスから算出する。

7. **一次正は `backend/prisma/schema.prisma`、docs/design.md の ER 図はそれと同期させる**
   - 根拠: CLAUDE.md のガードレールに「ドキュメントに先に書く → 派生作業」とあるため、設計変更時はまず design.md を更新し、次に Prisma スキーマに反映する運用とする。Prisma スキーマが実行可能な定義 (マイグレーション生成元) であるため、実装上の一次正は Prisma 側に置く。

## Consequences

### 利点

- エンティティが 2 つだけであり、CRUD の実装・テスト・デバッグが容易。
- Cascade 削除により、目的削除時の Todo 整合性が DB レベルで保証される。
- ステータスの DB 値と表示ラベルが一致しているため、フロントエンド側の変換コードが不要。
- 非正規化カラムがなく、データ整合性の維持コストが低い。

### 代償

- ステータスの選択肢が DB ではなくアプリケーションコードにハードコードされるため、値の追加・変更時にはコード修正 + デプロイが必要。ただし MVP スコープでは許容範囲。
- ステータスが String 型のため、DB レベルでの値制約がない。不正な値の混入はアプリケーションバリデーションに依存する。

### 将来のリスク

- マルチユーザ対応が必要になった場合、User エンティティの追加と Category/Todo への所有者 FK 追加が必要になる。現時点では未確定であり、発生時に ADR を起票して対応する。
- ステータスの選択肢をユーザがカスタマイズしたいという要件が発生した場合、StatusMaster テーブルへの移行が必要になる。現時点では未確定。

### 他の決定への波及

- [ADR-0002 (tech-stack)](0002-tech-stack.md) で決定した SQLite + Prisma の制約が、本 ADR でのステータス管理方式 (enum 不可 → String + アプリバリデーション) に直接影響している。
- `docs/design.md` セクション 2 (データモデル) および `docs/ui.md` セクション 3-5 (ステータスチップ) の記述は、本 ADR の Decision に基づいている。スキーマ変更時はこれらドキュメントの同期更新が必要。
