# 0004 全件表示機能の設計

## Status

Accepted

## Context

目的選択リストボックスに「(全て)」を追加し、全目的の Todo をまとめて表示する要望がある (US-007)。
以下の設計判断が必要:

1. 「(全て)」の表示/非表示を制御する設定の保存先
2. `selectedCategoryId` の状態表現方法
3. バックエンドの全件取得 API の実現方法
4. 全件表示時のテーブルレイアウト

## Decision

### 設定の保存先: localStorage

- DB にカラムを追加しない。ユーザーが自分だけのローカルアプリであり、UI プリファレンスは localStorage で十分。
- キー: `surulist:showAllEnabled` (名前空間プレフィックス付き)

### 状態表現: 文字列 `"all"` セントリネル

- `selectedCategoryId` の型を `number | "all" | null` に拡張。
- マジックナンバー `-1` は将来的なバグの温床になるため回避。`"all"` はコード上で自己文書化される。

### API: 既存 `GET /api/todos` の categoryId を optional 化

- `categoryId` パラメータ省略時に全 Todo を返却 (`include: { category: { select: { id, name } } }`)。
- 新規エンドポイントは作らない。既存の呼び出しは `categoryId` 指定済みのため後方互換。

### テーブルレイアウト: 単一テーブル + カテゴリ見出し行

- 複数テーブルではなく、`<tr colspan=6>` のグループ見出し行をカテゴリごとに挿入。
- 列幅の統一性が保たれ、実装も単純。

## Consequences

- **利点**: 全 Todo の横断的な確認が可能になる。設定は localStorage のため DB マイグレーション不要。
- **制約**: 全件表示中は新規追加ボタンを無効化する (追加先カテゴリが不定のため)。
- **将来**: Todo 件数が大量になった場合、全件取得のパフォーマンスが課題になりうる。ただし個人利用を前提としているため当面は問題ない。
